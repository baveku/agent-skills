# Review App

## Overview

Use this skill to produce a sharp consumer-app review focused on whether the app can be understood, shared, paid for, and scaled profitably.

Base the review on available evidence: app idea, screenshots, landing page, App Store listing, analytics, onboarding, paywall, retention, creator strategy, or user-acquisition data. If key evidence is missing, make explicit assumptions and ask only the highest-leverage follow-up question.

## Workflow

1. Identify the app's one job, target user, promised dream, and core pain.
2. Check whether the value is understood in 5 seconds and whether the app has a share-worthy magic moment.
3. Trace the first-session funnel: view/listing -> install -> onboarding -> paywall -> purchase -> first value.
4. Diagnose growth with the three primary metrics: view-to-install ratio, paywall reach rate, and paywall conversion rate.
5. Review paywall and onboarding as sales systems, not neutral UI flows.
6. Assess distribution: market tide, social proof, creator density, waitlist, manual outreach, and paid acquisition economics.
7. Separate current-stage priorities: pre-revenue, early revenue, scaling, or retention optimization.
8. Return a scorecard, top risks, and a short action plan ranked by business impact.

For deeper scoring criteria, read `references/viral_app_rubric.md`.

## Required Output

Keep the review concise and decisive. Use this structure unless the user asks for another format:

```markdown
## Verdict

[One paragraph: invest / test / pause / kill, with why.]

## Scorecard

| Area                        | Score | Diagnosis |
| --------------------------- | ----: | --------- |
| One-job clarity             |   /10 |           |
| Magic moment                |   /10 |           |
| Friction removal            |   /10 |           |
| 5-second comprehension      |   /10 |           |
| Paywall readiness           |   /10 |           |
| Onboarding sales power      |   /10 |           |
| Distribution potential      |   /10 |           |
| Monetization economics      |   /10 |           |
| Speed and operating cadence |   /10 |           |

## Biggest Leaks

[3-5 bullets, ordered by impact.]

## Next Moves

[3-5 concrete actions for the next 7-14 days.]
```

## Review Rules

- Prioritize willingness to pay over downloads, likes, or vague interest.
- Treat paywall performance as a core product signal, not a late-stage detail.
- Do not overvalue in-app polish before there is proof of acquisition and payment.
- Prefer deleting friction over adding features.
- Reward one strong wedge more than many adjacent features.
- Flag any strategy that requires slow consensus, long planning, or perfect launch conditions.
- When analytics are available, anchor the diagnosis in numbers before giving taste-based feedback.
- When analytics are missing, propose the smallest test that can reveal paid demand quickly.

## Metric Benchmarks

Use these as rough consumer-app targets, not universal laws:

- View-to-install: 5 installs per 1,000 views is workable.
- Paywall reach: 75% of installs reaching paywall is strong.
- Paywall conversion: 10% or higher is strong.
- Bad view-to-install usually means weak communication, weak market pull, or unclear dream.
- Good view-to-install with weak paywall conversion usually means the idea has pull but the product, offer, onboarding, price, or paywall is failing.

## Evidence Handling

If screenshots or app pages are provided, inspect them for actual user comprehension, hierarchy, friction, and sales clarity. If the user provides only an idea, review it as an investment memo and clearly label assumptions.

If the app is not a consumer subscription or viral utility app, adapt the rubric but state which principles may not apply.
