# Frontend UI Engineering

## Overview

Build production-quality user interfaces that are accessible, performant, and visually polished. The goal is UI that looks like it was built by a design-aware engineer at a top company — not like it was generated by an AI. This means real design system adherence, proper accessibility, thoughtful interaction patterns, and no generic "AI aesthetic."

This skill covers two platforms:

- **Web** — React, TypeScript, CSS/Tailwind
- **Apple** — Swift 6, SwiftUI, structured concurrency

## When to Use

- Building new UI components or pages
- Modifying existing user-facing interfaces
- Implementing responsive layouts
- Adding interactivity or state management
- Fixing visual or UX issues

## Component Architecture

### Web (React/TypeScript)

#### File Structure

Colocate everything related to a component:

```
src/components/
  TaskList/
    TaskList.tsx          # Component implementation
    TaskList.test.tsx     # Tests
    TaskList.stories.tsx  # Storybook stories (if using)
    use-task-list.ts      # Custom hook (if complex state)
    types.ts              # Component-specific types (if needed)
```

#### Component Patterns

**Prefer composition over configuration:**

```tsx
// Good: Composable
<Card>
  <CardHeader>
    <CardTitle>Tasks</CardTitle>
  </CardHeader>
  <CardBody>
    <TaskList tasks={tasks} />
  </CardBody>
</Card>

// Avoid: Over-configured
<Card
  title="Tasks"
  headerVariant="large"
  bodyPadding="md"
  content={<TaskList tasks={tasks} />}
/>
```

**Keep components focused:**

```tsx
// Good: Does one thing
export function TaskItem({ task, onToggle, onDelete }: TaskItemProps) {
  return (
    <li className="flex items-center gap-3 p-3">
      <Checkbox checked={task.done} onChange={() => onToggle(task.id)} />
      <span className={task.done ? 'line-through text-muted' : ''}>{task.title}</span>
      <Button variant="ghost" size="sm" onClick={() => onDelete(task.id)}>
        <TrashIcon />
      </Button>
    </li>
  );
}
```

**Separate data fetching from presentation:**

```tsx
// Container: handles data
export function TaskListContainer() {
  const { tasks, isLoading, error } = useTasks();

  if (isLoading) return <TaskListSkeleton />;
  if (error) return <ErrorState message="Failed to load tasks" retry={refetch} />;
  if (tasks.length === 0) return <EmptyState message="No tasks yet" />;

  return <TaskList tasks={tasks} />;
}

// Presentation: handles rendering
export function TaskList({ tasks }: { tasks: Task[] }) {
  return (
    <ul role="list" className="divide-y">
      {tasks.map(task => <TaskItem key={task.id} task={task} />)}
    </ul>
  );
}
```

#### Composition via ViewBuilder

**Prefer `@ViewBuilder` slots over configuration parameters:**

```swift
// Good: Composable slots
struct CardView<Header: View, Content: View>: View {
    let header: Header
    let content: Content

    init(
        @ViewBuilder header: () -> Header,
        @ViewBuilder content: () -> Content
    ) {
        self.header = header()
        self.content = content()
    }

    var body: some View {
        VStack(alignment: .leading, spacing: 0) {
            header
                .padding()
            Divider()
            content
                .padding()
        }
        .background(.background.secondary)
        .clipShape(RoundedRectangle(cornerRadius: 12))
    }
}

// Usage
CardView {
    Text("Tasks")
        .font(.headline)
} content: {
    TaskListView(tasks: tasks)
}
```

#### Extract Subviews

**Keep views under ~80 lines. Extract early, not late:**

```swift
// Good: Focused subview
struct TaskItemView: View {
    let task: TaskItem
    var onToggle: () -> Void
    var onDelete: () -> Void

    var body: some View {
        HStack(spacing: 12) {
            Toggle(isOn: .init(
                get: { task.isDone },
                set: { _ in onToggle() }
            )) {
                Text(task.title)
                    .strikethrough(task.isDone)
                    .foregroundStyle(task.isDone ? .secondary : .primary)
            }
            .toggleStyle(.checkbox)

            Spacer()

            Button(role: .destructive, action: onDelete) {
                Image(systemName: "trash")
            }
            .buttonStyle(.borderless)
        }
        .padding(.vertical, 8)
    }
}
```

#### ViewModel + View Split

**Separate logic from layout. Use `@Observable` (not ObservableObject):**

```swift
import SwiftUI

@Observable
final class TaskListViewModel {
    private(set) var tasks: [TaskItem] = []
    private(set) var isLoading = false
    var error: Error?

    private let repository: TaskRepository

    init(repository: TaskRepository = .shared) {
        self.repository = repository
    }

    func loadTasks() async {
        isLoading = true
        defer { isLoading = false }
        do {
            tasks = try await repository.fetchAll()
        } catch {
            self.error = error
        }
    }

    func toggleTask(_ task: TaskItem) async {
        // Optimistic update
        if let index = tasks.firstIndex(where: { $0.id == task.id }) {
            tasks[index].isDone.toggle()
        }
        do {
            try await repository.toggle(task)
        } catch {
            // Revert on failure
            if let index = tasks.firstIndex(where: { $0.id == task.id }) {
                tasks[index].isDone.toggle()
            }
            self.error = error
        }
    }
}

// View consumes the ViewModel
struct TaskListView: View {
    @State private var viewModel = TaskListViewModel()

    var body: some View {
        List {
            ForEach(viewModel.tasks) { task in
                TaskItemView(
                    task: task,
                    onToggle: { Task { await viewModel.toggleTask(task) } },
                    onDelete: { Task { await viewModel.deleteTask(task) } }
                )
            }
        }
        .task { await viewModel.loadTasks() }
    }
}
```

## State Management

### Web (React/TypeScript)

**Choose the simplest approach that works:**

```
Local state (useState)           → Component-specific UI state
Lifted state                     → Shared between 2-3 sibling components
Context                          → Theme, auth, locale (read-heavy, write-rare)
URL state (searchParams)         → Filters, pagination, shareable UI state
Server state (React Query, SWR)  → Remote data with caching
Global store (Zustand, Redux)    → Complex client state shared app-wide
```

**Avoid prop drilling deeper than 3 levels.** If you're passing props through components that don't use them, introduce context or restructure the component tree.

### Apple (Swift/SwiftUI)

**Choose the simplest property wrapper that fits:**

```
@State                → View-local, value-type UI state (toggles, text fields, sheet presentation)
@Binding              → Child view writes back to parent's @State
@Observable + @State  → View-local reference-type state (view models)
@Bindable             → Create Bindings from @Observable properties
@Environment          → Injected dependencies (\.modelContext, \.dismiss, custom keys)
@Query (SwiftData)    → Reactive model fetches, sorted/filtered
@AppStorage           → Small UserDefaults-backed values (settings, onboarding flags)
```

**Example — passing state correctly:**

```swift
// Parent owns the source of truth
struct TaskListView: View {
    @State private var selectedTask: TaskItem?
    @State private var isAddingTask = false

    var body: some View {
        List(tasks, selection: $selectedTask) { task in
            TaskItemView(task: task)
        }
        .sheet(isPresented: $isAddingTask) {
            AddTaskView() // no binding needed if it saves via SwiftData
        }
    }
}

// Child receives binding when it needs to write back
struct FilterBar: View {
    @Binding var selectedCategory: Category?

    var body: some View {
        Picker("Category", selection: $selectedCategory) {
            Text("All").tag(Category?.none)
            ForEach(Category.allCases) { cat in
                Text(cat.displayName).tag(Optional(cat))
            }
        }
        .pickerStyle(.segmented)
    }
}
```

**Using @Observable with @Environment for dependency injection:**

```swift
// Define an environment key
struct TaskRepositoryKey: EnvironmentKey {
    static let defaultValue: TaskRepository = .shared
}

extension EnvironmentValues {
    var taskRepository: TaskRepository {
        get { self[TaskRepositoryKey.self] }
        set { self[TaskRepositoryKey.self] = newValue }
    }
}

// Inject via the environment
struct TaskListView: View {
    @Environment(\.taskRepository) private var repository
    @State private var viewModel: TaskListViewModel?

    var body: some View {
        Group {
            if let viewModel {
                TaskListContent(viewModel: viewModel)
            } else {
                ProgressView()
            }
        }
        .onAppear {
            viewModel = TaskListViewModel(repository: repository)
        }
    }
}
```

**Using SwiftData @Query for reactive lists:**

```swift
import SwiftData

@Model
final class TaskItem {
    var title: String
    var isDone: Bool
    var createdAt: Date

    init(title: String, isDone: Bool = false) {
        self.title = title
        self.isDone = isDone
        self.createdAt = .now
    }
}

struct TaskListView: View {
    @Query(sort: \TaskItem.createdAt, order: .reverse)
    private var tasks: [TaskItem]

    @Environment(\.modelContext) private var modelContext

    var body: some View {
        List {
            ForEach(tasks) { task in
                TaskItemView(task: task)
            }
            .onDelete { indexSet in
                for index in indexSet {
                    modelContext.delete(tasks[index])
                }
            }
        }
    }
}
```

## Design System Adherence

### Avoid the AI Aesthetic

AI-generated UI has recognizable patterns. Avoid all of them:

| AI Default | Why It Is a Problem | Production Quality |
|---|---|---|
| Purple/indigo everything | Models default to visually "safe" palettes, making every app look identical | Use the project's actual color palette |
| Excessive gradients | Gradients add visual noise and clash with most design systems | Flat or subtle gradients matching the design system |
| Rounded everything (rounded-2xl) | Maximum rounding signals "friendly" but ignores the hierarchy of corner radii in real designs | Consistent border-radius from the design system |
| Generic hero sections | Template-driven layout with no connection to the actual content or user need | Content-first layouts |
| Lorem ipsum-style copy | Placeholder text hides layout problems that real content reveals (length, wrapping, overflow) | Realistic placeholder content |
| Oversized padding everywhere | Equal generous padding destroys visual hierarchy and wastes screen space | Consistent spacing scale |
| Stock card grids | Uniform grids are a layout shortcut that ignores information priority and scanning patterns | Purpose-driven layouts |
| Shadow-heavy design | Layered shadows add depth that competes with content and slows rendering on low-end devices | Subtle or no shadows unless the design system specifies |

### Web (React/TypeScript)

#### Spacing and Layout

Use a consistent spacing scale. Don't invent values:

```css
/* Use the scale: 0.25rem increments (or whatever the project uses) */
/* Good */  padding: 1rem;      /* 16px */
/* Good */  gap: 0.75rem;       /* 12px */
/* Bad */   padding: 13px;      /* Not on any scale */
/* Bad */   margin-top: 2.3rem; /* Not on any scale */
```

#### Typography

Respect the type hierarchy:

```
h1 → Page title (one per page)
h2 → Section title
h3 → Subsection title
body → Default text
small → Secondary/helper text
```

Don't skip heading levels. Don't use heading styles for non-heading content.

#### Color

- Use semantic color tokens: `text-primary`, `bg-surface`, `border-default` — not raw hex values
- Ensure sufficient contrast (4.5:1 for normal text, 3:1 for large text)
- Don't rely solely on color to convey information (use icons, text, or patterns too)

### Apple (Swift/SwiftUI)

#### Custom ViewModifiers

**Encapsulate repeated styling into reusable modifiers:**

```swift
struct CardModifier: ViewModifier {
    func body(content: Content) -> some View {
        content
            .padding()
            .background(.background.secondary)
            .clipShape(RoundedRectangle(cornerRadius: 12))
            .shadow(color: .black.opacity(0.05), radius: 4, y: 2)
    }
}

extension View {
    func cardStyle() -> some View {
        modifier(CardModifier())
    }
}

// Usage
VStack { ... }
    .cardStyle()
```

#### Custom ButtonStyle and TextFieldStyle

**Define styles once, apply everywhere:**

```swift
struct PrimaryButtonStyle: ButtonStyle {
    @Environment(\.isEnabled) private var isEnabled

    func makeBody(configuration: Configuration) -> some View {
        configuration.label
            .font(.body.weight(.semibold))
            .foregroundStyle(.white)
            .padding(.horizontal, 24)
            .padding(.vertical, 12)
            .background(
                isEnabled
                    ? Color.accentColor
                    : Color.accentColor.opacity(0.4),
                in: RoundedRectangle(cornerRadius: 10)
            )
            .opacity(configuration.isPressed ? 0.8 : 1.0)
            .animation(.easeInOut(duration: 0.15), value: configuration.isPressed)
    }
}

struct AppTextFieldStyle: TextFieldStyle {
    func _body(configuration: TextField<Self._Label>) -> some View {
        configuration
            .padding(12)
            .background(.background.secondary)
            .clipShape(RoundedRectangle(cornerRadius: 8))
            .overlay(
                RoundedRectangle(cornerRadius: 8)
                    .strokeBorder(.quaternary)
            )
    }
}

// Apply globally or per-view
TextField("Task title", text: $title)
    .textFieldStyle(AppTextFieldStyle())

Button("Save") { save() }
    .buttonStyle(PrimaryButtonStyle())
```

#### Spacing, Typography, and Color

**Use Apple's built-in semantic values. Avoid magic numbers:**

```swift
// Spacing: use standard padding values
.padding()                    // System default (16pt)
.padding(.horizontal, 20)    // Acceptable custom value
.padding(13)                  // ✗ Not on any standard scale

// Typography: use the Dynamic Type scale
.font(.title)                 // ✓ Adapts to user's text size preference
.font(.headline)              // ✓
.font(.system(size: 17))      // ✗ Hardcoded — breaks Dynamic Type

// Color: use semantic colors
.foregroundStyle(.primary)    // ✓ Adapts to light/dark mode
.foregroundStyle(.secondary)  // ✓
.background(.background)      // ✓
.foregroundColor(Color(red: 0.2, green: 0.3, blue: 0.8))  // ✗ Raw values

// Custom semantic colors via Asset Catalog or extension
extension ShapeStyle where Self == Color {
    static var brandAccent: Color { Color("BrandAccent") }
}
```

## Accessibility

### Web (React/TypeScript)

Every component must meet WCAG 2.1 AA standards.

#### Keyboard Navigation

```tsx
// Every interactive element must be keyboard accessible
<button onClick={handleClick}>Click me</button>        // ✓ Focusable by default
<div onClick={handleClick}>Click me</div>               // ✗ Not focusable
<div role="button" tabIndex={0} onClick={handleClick}    // ✓ But prefer <button>
     onKeyDown={e => {
       if (e.key === 'Enter') handleClick();
       if (e.key === ' ') e.preventDefault();
     }}
     onKeyUp={e => {
       if (e.key === ' ') handleClick();
     }}>
  Click me
</div>
```

#### ARIA Labels

```tsx
// Label interactive elements that lack visible text
<button aria-label="Close dialog"><XIcon /></button>

// Label form inputs
<label htmlFor="email">Email</label>
<input id="email" type="email" />

// Or use aria-label when no visible label exists
<input aria-label="Search tasks" type="search" />
```

#### Focus Management

```tsx
// Move focus when content changes
function Dialog({ isOpen, onClose }: DialogProps) {
  const closeRef = useRef<HTMLButtonElement>(null);

  useEffect(() => {
    if (isOpen) closeRef.current?.focus();
  }, [isOpen]);

  // Trap focus inside dialog when open
  return (
    <dialog open={isOpen}>
      <button ref={closeRef} onClick={onClose}>Close</button>
      {/* dialog content */}
    </dialog>
  );
}
```

### Apple (Swift/SwiftUI)

Every view must support VoiceOver, Dynamic Type, and keyboard/Switch Control navigation.

#### Accessibility Labels and Hints

```swift
// Label elements that lack visible text
Button(action: deleteTask) {
    Image(systemName: "trash")
}
.accessibilityLabel("Delete task")
.accessibilityHint("Removes this task permanently")

// Combine child elements into a single accessible unit
HStack {
    Image(systemName: task.isDone ? "checkmark.circle.fill" : "circle")
    Text(task.title)
    Spacer()
    Text(task.dueDate, style: .date)
}
.accessibilityElement(children: .combine)
.accessibilityLabel("\(task.title), due \(task.dueDate.formatted())")
.accessibilityAddTraits(task.isDone ? .isSelected : [])
```

#### Focus Management with @FocusState

```swift
struct AddTaskView: View {
    @State private var title = ""
    @State private var notes = ""
    @FocusState private var focusedField: Field?

    enum Field: Hashable {
        case title, notes
    }

    var body: some View {
        Form {
            TextField("Title", text: $title)
                .focused($focusedField, equals: .title)

            TextField("Notes", text: $notes)
                .focused($focusedField, equals: .notes)
        }
        .onAppear { focusedField = .title }
        .onSubmit {
            switch focusedField {
            case .title: focusedField = .notes
            case .notes: save()
            case nil: break
            }
        }
    }
}
```

#### VoiceOver Best Practices

```swift
// Custom actions for swipe gestures
TaskItemView(task: task)
    .accessibilityAction(named: "Toggle completion") {
        viewModel.toggleTask(task)
    }
    .accessibilityAction(named: "Delete") {
        viewModel.deleteTask(task)
    }

// Announce changes
Button("Save") {
    save()
    AccessibilityNotification.Announcement("Task saved")
        .post()
}

// Hide decorative elements
Image(systemName: "chevron.right")
    .accessibilityHidden(true)
```

## Loading and Transitions

### Web (React/TypeScript)

```tsx
// Skeleton loading (not spinners for content)
function TaskListSkeleton() {
  return (
    <div className="space-y-3" aria-busy="true" aria-label="Loading tasks">
      {Array.from({ length: 3 }).map((_, i) => (
        <div key={i} className="h-12 bg-muted animate-pulse rounded" />
      ))}
    </div>
  );
}

// Optimistic updates for perceived speed
function useToggleTask() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: toggleTask,
    onMutate: async (taskId) => {
      await queryClient.cancelQueries({ queryKey: ['tasks'] });
      const previous = queryClient.getQueryData(['tasks']);

      queryClient.setQueryData(['tasks'], (old: Task[]) =>
        old.map(t => t.id === taskId ? { ...t, done: !t.done } : t)
      );

      return { previous };
    },
    onError: (_err, _taskId, context) => {
      queryClient.setQueryData(['tasks'], context?.previous);
    },
  });
}
```

### Apple (Swift/SwiftUI)

**Use `ProgressView` and `.redacted(reason:)` for loading states:**

```swift
struct TaskListView: View {
    @State private var viewModel = TaskListViewModel()

    var body: some View {
        Group {
            if viewModel.isLoading && viewModel.tasks.isEmpty {
                // Skeleton / redacted placeholder
                List(TaskItem.placeholders) { task in
                    TaskItemView(task: task)
                        .redacted(reason: .placeholder)
                }
                .disabled(true)
            } else {
                List(viewModel.tasks) { task in
                    TaskItemView(task: task)
                }
            }
        }
        .overlay {
            if viewModel.isLoading && !viewModel.tasks.isEmpty {
                // Inline refresh indicator for subsequent loads
                ProgressView()
                    .frame(maxWidth: .infinity, maxHeight: .infinity, alignment: .top)
                    .padding(.top, 8)
            }
        }
        .task { await viewModel.loadTasks() }
        .refreshable { await viewModel.loadTasks() }
    }
}
```

**Placeholder data for redacted skeletons:**

```swift
extension TaskItem {
    static let placeholders: [TaskItem] = (0..<5).map { i in
        TaskItem(title: String(repeating: "A", count: Int.random(in: 15...30)))
    }
}
```

**Animated transitions:**

```swift
// Smooth insertion / removal
ForEach(viewModel.tasks) { task in
    TaskItemView(task: task)
        .transition(.asymmetric(
            insertion: .move(edge: .top).combined(with: .opacity),
            removal: .move(edge: .trailing).combined(with: .opacity)
        ))
}
```

## Responsive Design

### Web (React/TypeScript)

Design for mobile first, then expand:

```tsx
// Tailwind: mobile-first responsive
<div className="
  grid grid-cols-1      /* Mobile: single column */
  sm:grid-cols-2        /* Small: 2 columns */
  lg:grid-cols-3        /* Large: 3 columns */
  gap-4
">
```

Test at these breakpoints: 320px, 768px, 1024px, 1440px.

### Apple (Swift/SwiftUI)

SwiftUI adapts across iPhone, iPad, and Mac via size classes and platform idioms. Prefer semantic layouts that adapt automatically:

```swift

                TaskListView()
```

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "Accessibility is a nice-to-have" | It's a legal requirement in many jurisdictions and an engineering quality standard. |
| "We'll make it responsive later" | Retrofitting responsive design is 3x harder than building it from the start. |
| "The design isn't final, so I'll skip styling" | Use the design system defaults. Unstyled UI creates a broken first impression for reviewers. |
| "This is just a prototype" | Prototypes become production code. Build the foundation right. |
| "The AI aesthetic is fine for now" | It signals low quality. Use the project's actual design system from the start. |
| "SwiftUI handles accessibility automatically" | It provides a baseline, but custom views, combined elements, and meaningful labels still need manual work. |
| "I'll add navigation later" | Navigation architecture is structural. Retrofitting `NavigationStack` paths is significantly harder than designing them upfront. |

## Red Flags

### Web (React/TypeScript)

- Components with more than 200 lines (split them)
- Inline styles or arbitrary pixel values
- Missing error states, loading states, or empty states
- No keyboard navigation testing
- Color as the sole indicator of state (red/green without text or icons)
- Generic "AI look" (purple gradients, oversized cards, stock layouts)

### Apple (Swift/SwiftUI)

- Views exceeding ~80 lines without extracting subviews
- Using `ObservableObject` / `@Published` instead of `@Observable` (Swift 5.9+)
- Hardcoded `.font(.system(size:))` instead of Dynamic Type (`.font(.body)`)
- Raw color literals instead of semantic colors (`.primary`, `.secondary`, Asset Catalog)
- Force-unwrapping optionals in view code (`task!.title`)
- Missing `.accessibilityLabel()` on icon-only buttons
- Using `@State` for reference types or complex objects (use `@State` with `@Observable`)
- Empty `List` or blank screen with no `ContentUnavailableView`
- Navigation via `@State` booleans for every destination instead of `NavigationPath`
- Magic-number padding (`.padding(13)`) instead of standard values

## Verification

After building UI:

### Web (React/TypeScript)

- [ ] Component renders without console errors
- [ ] All interactive elements are keyboard accessible (Tab through the page)
- [ ] Screen reader can convey the page's content and structure
- [ ] Responsive: works at 320px, 768px, 1024px, 1440px
- [ ] Loading, error, and empty states all handled
- [ ] Follows the project's design system (spacing, colors, typography)
- [ ] No accessibility warnings in dev tools or axe-core

### Apple (Swift/SwiftUI)

- [ ] Builds without warnings on latest Xcode
- [ ] VoiceOver reads every screen correctly — enable it and navigate every view
- [ ] Dynamic Type: test at default, largest accessibility, and smallest sizes
- [ ] Light mode and dark mode both render correctly
- [ ] Loading state shown (ProgressView or .redacted) while data is fetching
- [ ] Error state shown via ContentUnavailableView with retry action
- [ ] Empty state shown via ContentUnavailableView with call-to-action
- [ ] Navigation works via NavigationStack; back button and swipe-back both function
- [ ] No layout breakage on iPhone SE, standard iPhone, iPad
- [ ] Previews render in Xcode without crashing
- [ ] Keyboard and Switch Control can reach all interactive elements
